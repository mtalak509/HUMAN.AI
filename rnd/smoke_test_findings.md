# Smoke-test findings — LLM-экстрактор резюме

*Версия 1.0. Итог нулевой итерации R&D по плану `rnd/plan/rnd_extractor_plan.md` v0.3. Дата прогона: 2026-05-15.*

---

## 1. TL;DR

**Развилка по §11 плана — зелёный.** Подход (PyPDF2 + text-only LLM) на корпусе из 5 резюме даёт валидные `Resume` JSON'ы без систематических галлюцинаций. Разворачиваем полноценный R&D.

**Дефолтный режим для следующей итерации — `response_format=json_object` + Pydantic-валидация + 1 retry.** Альтернатива `response_format=json_schema` показала набор отличий (плюсов и минусов) и держится как отдельная гипотеза.

## 5. Что работает (главное)

1. **Подход в принципе валиден.** PyPDF2 справляется со всеми 5 PDF, LLM получает чистый text и возвращает JSON, проходящий Pydantic-валидацию. Ни одного полного фейла, ни одного retry'я по `ValidationError`.
2. **Английский язык работает наравне с русским** (Markova). Это снимает беспокойство, что промпт «русоцентричен».
3. **Никаких выдуманных описаний.** На Markova, где в исходнике нет описаний должностей, оба режима корректно вернули `description = null` и `skills_mentioned = []`, а не нагенерили правдоподобную выдумку. Анти-галлюцинаторное поведение по описаниям — устойчивое.
4. **Даты, компании, роли — извлекаются точно** на всех 5 резюме обоих режимов.
5. **Контакты** распознаются по типу (`phone` / `email` / `telegram` / `linkedin`) корректно.

## 6. Системные находки про `json_object` vs `json_schema`

### 6.1 Неполный сводный `skills` у `json_schema` — баг режима (4-5/5)

Самое надёжное наблюдение. Навыки, которые модель сама же подняла в `experiences[*].skills_mentioned`, систематически **не попадают** в сводный список `skills`.

Примеры:
- **Talakin**: пропали `R-CNN`, `Faster R-CNN`, `CNN`, `Управление проектами`, `НИОКР` (хотя они в `skills_mentioned` ролей).
- **Talakina**: пропали `управление командой`, `1-1`, `мониторинг рынка труда`.
- **Denisenko**: пропали `анализ дебиторской задолженности`, `организация мероприятий`, `управление календарем`.
- **Suhanova**: пропали **все 4 языка** (`Русский`, `Английский B2`, `Испанский A1`, `Сербский A2`) — это уже теряются целые блоки.
- **Markova**: размытее (выкинуты `KVs`, `TVCs`, `Fans Operations`, но нормализованы `SN`→`Social Networks`).

Прямое нарушение правила промпта *«Сводный список skills — все навыки, которые встречаются в резюме»*.

**Импликация для R&D:** если использовать `json_schema` в проде — обязательный post-processing вида `skills ∪= union(experiences[*].skills_mentioned)`.

### 6.2 Фактические ошибки у `json_schema` (2/5)

- **Talakin**: `experiences[0].to_date = "2025-12"` на текущей работе (правильно — `null`). Галлюцинация даты, прямое нарушение правила промпта *«Для текущего места работы to_date = null»*.
- **Suhanova**: `education[1].degree = "Высшее"` для записи «Колледж РГУТС» (правильно — «Среднее профессиональное»). Неверная типизация существующего, критичная для downstream-фильтрации.

В `json_object` таких ошибок **0/5**. Это сильный аргумент в пользу parsed-режима для нашего домена.

### 6.3 Мини-галлюцинации в `skills_mentioned` у `json_schema` (1/5)

- **Suhanova**, GC Group: в `description` буквально нет «бронирование отелей», «бронирование авиабилетов», «оформление виз» — только «полный спектр услуг по travel поддержке». А в `skills_mentioned` они появились. Интерпретация, не извлечение.

В `parsed`-режиме на той же роли — `travel поддержка` без разбивки. То есть `parsed` остаётся ближе к тексту, `structured` склонен «доинтерпретировать».

### 6.4 Где `json_schema` лучше

- **Локальная конкретика**: «(Россия, Бразилия, Мексика)» у Talakina, «логистика, продажи, склад» у Denisenko, atomic `allotment` / `commitment` у Suhanova.
- **Нормализация жаргона** (Markova): `SN` → `Social Networks`, `Community` → `Community Management`, `E-com Strategy` → `E-commerce`. Для retrieval по «человеческим» формулировкам — плюс.
- **Гранулярность `skills_mentioned`**: разбивает склейки вроде «transfer/rooming list» на atomic-навыки. Для графа полезно.

### 6.5 Длина `description` — несистемно, зависит от стиля

Гипотеза «structured короче» подтвердилась на Talakin/Talakina (повествовательные тексты) и опроверглась на Denisenko/Suhanova (буллетные резюме — structured там подробнее). На Markova вопрос не стоит. Итого — режим тут не главный фактор, главное структура исходника.

### 6.6 Непоследовательная нормализация (шум, не баг режима)

Поле `education[*].institution` — на Talakin `structured` добавлял «, Москва», на Talakina наоборот добавил `parsed`. Модель ведёт себя непоследовательно от документа к документу в плане нормализации. На это поле нельзя полагаться без отдельного post-processing шага.

Бюджет уложился в план (~5 $). По логам `usage` — порядок ~5-15K total_tokens на резюме на каждый прогон, два режима × 5 резюме = 10 вызовов. Запас огромный.

## 8. Решение

**Зелёный по §11.** Разворачиваем полноценный R&D.

### Что забираем в полноценный R&D из этого спринта

1. **Базовая архитектура подтверждена**: `PyPDF2 → text → LLM (text-only) → JSON → Pydantic`. Без OCR-fallback'а (пока).
2. **Дефолтный режим — `response_format=json_object` + Pydantic + 1 retry.** Ни одной ошибки на 5 резюме, понятный fallback на retry.
3. **Pydantic-схема `Resume`** в текущем виде работает; ничего не пришлось добавлять или резать.
4. **Промпт zero-shot, температура 0** — достаточно для качественного извлечения на текущих 5 резюме.

### Что добавить в полноценный R&D

1. **Корпус 30 резюме** с явным сегментом по стилю:
   - 8 повествовательных (HH-выгрузки IT/ML),
   - 8 буллетных (ассистенты, HR),
   - 6 executive (короткие, английские/билингвальные),
   - 4 «творческих» (двухколоночные, с графикой),
   - 4 скан-PDF (image-only).
2. **3 модели**: Qwen3, GPT-4o-mini, Claude Haiku (или эквивалент через OpenRouter). Запуск каждой и сравнение.
3. **Сравнение режимов**: `json_object` vs `json_schema` с явным post-processing для skills у второго.
4. **Skip-условие для скан-PDF**: если `extract_text()` пустой на странице — fallback на `pdfplumber` или `PyMuPDF`, и если те тоже не справляются — фиксируем как ограничение подхода и считаем долю «не извлекаемых» резюме отдельной метрикой.
5. **Golden set** — разметка вручную на 30 резюме (приоритет: `experiences` даты/компании/роли, `education.degree`, контакты).
6. **Эвал-харнес** с метриками по типам полей:
   - Контакты: precision/recall.
   - Experiences: F1 по матчингу (компания + роль + диапазон дат), отдельно — точность дат.
   - Education: точность `degree`-типа (категориальная классификация).
   - Skills (сводный): recall vs golden-список.
   - **Отдельный bucket «фактические ошибки»** (галлюцинация даты, неверный degree-тип) — критичные для графа.
   - **Отдельный bucket «мини-галлюцинации в skills_mentioned»** — навыки, которых нет в тексте описания.

### Что НЕ делать в полноценном R&D (явные границы)

- Не пытаться чинить промпт в этом спринте под `structured`-режим. Сначала измерить, потом править.
- Не лезть в multimodal (Qwen3-VL) — `parsed`-подход на текстовых PDF работает, multimodal оставлен как fallback для скан-категории, не как дефолт.
- Не делать стриминг / batch API — на 30 резюме это лишнее, последовательного синхронного вызова достаточно.

## 9. Известные ограничения этого smoke-теста

1. **Не тестировались сканы** — все 5 PDF имели извлекаемый текст. Реальное поведение PyPDF2 на сканах — TBD в полноценном R&D.
2. **Не тестировались «творческие» PDF** (с тяжёлым layout'ом, графикой, двухколоночные). Markova была частично сложной (короткий executive CV), но всё же читалась ровно.
3. **Одна модель, один промпт, одна температура.** Это и было целью smoke-теста — но и обобщать findings на «LLM в целом» нельзя.
4. **5 резюме — статистика мала.** Все наблюдения этого документа — гипотезы для проверки на 30 резюме, не финальные выводы.
5. **Конфликт с самим собой**: `structured`-режим в одном и том же JSON генерирует `skills_mentioned`, не консистентные со сводным `skills`. Это не фиксилось retry-ом (его и не было), и в полноценном R&D надо решать post-processing'ом или другим промптом.
