# R&D-спринт: smoke-test LLM-экстрактора

*Версия 0.3. Минимальная проверка концепции на 5 резюме через text-only LLM по OpenRouter. PDF → текст напрямую (PyPDF2) → LLM. Это не go/no-go по Г1 из `core_architecture.md`, а нулевая итерация: посмотреть, на что вообще способен подход. Полноценный R&D с замером метрик — отдельно, если smoke-test даст зелёный сигнал.*

**Изменение vs 0.2:** отказались от мультимодальной подачи PDF как изображений. Вместо растеризации в PNG и Qwen3-VL — прямое извлечение текста из PDF через PyPDF2 и передача plain text в обычную (не multimodal) LLM. Причины: дешевле (нет картинок в контексте), быстрее, проще пайплайн, и это ближе к продакшен-варианту, где растеризация всех резюме экономически не оправдана.

---

## 1. Цель

Ответить на один вопрос: **способна ли text-only LLM, получая на вход текст резюме, извлечённый напрямую из PDF через PyPDF2, вернуть валидно структурированную информацию о кандидате в нашей Pydantic-схеме?**

Параллельный под-вопрос: **достаточно ли хорошо PyPDF2 извлекает текст из реальных PDF-резюме?** Качество парсинга — половина успеха; если текст приходит ломаный (склеенные колонки, потерянный layout, мусор), это нужно увидеть заранее.

Это «работает / не работает», без точных метрик. На 5 резюме статистика всё равно шумная, и не в ней смысл — смысл в том, чтобы получить руками потрогать выход модели и понять, в какую сторону копать дальше.

## 3. Корпус

**5 резюме в PDF.** Распределение:
- 3 шт. — обычные HH-выгрузки или Word-резюме.
- 1 шт. — самописный PDF с нестандартным layout.
- 1 шт. — «творческое» (длинное, двухколоночное, с визуальными элементами).

DOCX в этом спринте не используем — упрощаем подачу до одного формата.

Источник — публичные резюме с HH или личные знакомые с разрешением. Обезличивать не обязательно, файлы локально и нигде не публикуются.

## 4. Extraction schema

Pydantic v2, минимальный набор:

```python
from pydantic import BaseModel, Field
from typing import Optional, Literal

class Contact(BaseModel):
    type: Literal["email", "phone", "telegram", "linkedin", "other"]
    value: str

class Experience(BaseModel):
    from_date: str = Field(description="YYYY-MM или YYYY")
    to_date: Optional[str] = Field(description="None если по настоящее время")
    company: str
    role: str
    description: Optional[str]
    skills_mentioned: list[str]

class Education(BaseModel):
    institution: str
    degree: Optional[str]
    field: Optional[str]
    from_date: Optional[str]
    to_date: Optional[str]

class Resume(BaseModel):
    full_name: str
    contacts: list[Contact]
    experiences: list[Experience]
    education: list[Education]
    skills: list[str]
```

## 5. Модель и подача документа

**Модель:** text-only через OpenRouter — кандидаты на старт: `qwen/qwen3-...` или `openai/gpt-4o-mini` (точное имя — на момент имплементации, OpenRouter совместим с OpenAI Chat Completions API). Берём одну, не сравниваем.

**Подача PDF:**
1. PDF → plain text через `PyPDF2.PdfReader` постранично.
2. Тексты страниц склеиваются с разделителем `\n\n--- PAGE N ---\n\n` для сохранения границ.
3. Полученный текст подставляется в промпт как обычное сообщение `role="user"`.

```python
from pypdf import PdfReader  # PyPDF2 в современной упаковке = pypdf

def pdf_to_text(path: str) -> str:
    reader = PdfReader(path)
    pages = []
    for i, page in enumerate(reader.pages, 1):
        pages.append(f"--- PAGE {i} ---\n{page.extract_text() or ''}")
    return "\n\n".join(pages)

content = [
    {"type": "text", "text": PROMPT.format(resume_text=pdf_to_text(path))}
]
```

**Обработка пограничных случаев парсинга:**
- Пустой `extract_text()` на странице (скан/картинка) — допускаем, в findings помечаем такие резюме отдельно. Smoke-test не пытается решить проблему сканов; если в корпусе попадётся такой PDF, фиксируем как ограничение подхода.
- Без OCR fallback'а в этом спринте. Если все 5 резюме окажутся «текстовыми» — отлично; если попадётся скан — это сигнал в findings, а не повод доделывать пайплайн.

**Настройки:** `temperature=0`. Structured output — пробуем через `response_format={"type": "json_schema", ...}` если модель поддерживает; fallback на JSON mode + Pydantic-валидация с одним retry на невалидном выходе.

## 6. Промпт

Один промпт, zero-shot:

```
Ты извлекаешь структурированную информацию из резюме кандидата.
На входе — текст резюме, извлечённый из PDF постранично. Границы страниц
помечены маркерами `--- PAGE N ---`. Текст может содержать артефакты парсинга
(склеенные слова, разорванные строки, перепутанный порядок колонок) —
интерпретируй смысл, не цепляйся за форматирование.

Заполни поля JSON-схемы согласно содержимому.
Правила:
- Если поле не упомянуто в резюме — null.
- Не выдумывай данные, которых нет в тексте.
- Для текущего места работы to_date = null.
- skills_mentioned для каждой роли — только навыки, явно упомянутые
  в описании этой конкретной роли (не сводный список).
- Сводный список skills — все навыки, которые встречаются в резюме.

Текст резюме:
---
{resume_text}
---
```

## 7. Что проверяем глазами

На 5 резюме автоматические метрики бессмысленны. Вместо них — чек-лист на каждое резюме:

- [ ] Ответ модели — валидный JSON, парсится в `Resume`.
- [ ] `full_name` совпадает с резюме.
- [ ] Все контакты из резюме присутствуют в `contacts`.
- [ ] Количество `experiences` совпадает с резюме (±1).
- [ ] У каждого experience корректные даты, компания, роль.
- [ ] `description` не пустой там, где он был в резюме.
- [ ] `skills_mentioned` для каждой роли — реально упомянутые, не выдуманные.
- [ ] `skills` (сводный) разумный по объёму.
- [ ] Нет галлюцинаций (выдуманные навыки, компании, периоды).
- [ ] Промежуточный текст после PyPDF2 — читаемый (визуально пробежать `*.txt` в `results/`). Отметить, если PDF — скан или layout развалился настолько, что текста почти нет.

Параллельно — для каждого резюме записываем стоимость в токенах и время отклика.

## 8. Pipeline

### День 1: подготовка (2–3 часа)
- Зарегистрироваться на OpenRouter, получить API-ключ, положить 5–10 $ на счёт.
- Собрать 5 PDF-резюме.
- Создать структуру проекта (минимальную, см. §9).
- Реализовать `pdf_to_text()` (PyPDF2 / pypdf) и проверить на одном резюме — глазами посмотреть, что извлеклось.

### День 2: имплементация и прогон (3–4 часа)
- Pydantic-схема.
- OpenRouter-клиент с text-only подачей.
- Промпт.
- Прогон 5 резюме, сохранить выходы в `results/`.

### День 3: анализ (2–3 часа)
- Пройти чек-лист по каждому из 5 резюме.
- Записать findings в `smoke_test_findings.md`: что работает, что нет, какие категории ошибок встречаются.
- Принять решение по развилке (§11).

## 11. Развилки

### Зелёный: всё в целом работает
По чек-листу 80%+ галочек на большинстве резюме, нет систематических галлюцинаций. **Действия:** разворачиваем полноценный R&D (30 резюме, 3 модели, метрики) — у нас есть подтверждение, что копать стоит.

### Жёлтый: работает на простых, ломается на сложном
Простые HH-резюме читаются, творческие — нет. **Действия:** добавляем в полноценный R&D сегмент по типу резюме, проверяем, не помогает ли смена промпта на двухпроходный (сегментация → экстракция). Сам подход не отбрасываем.

### Красный: галлюцинации, schema compliance < 80%, или просто бред
Text-only LLM не справляется. Прежде чем хоронить подход, нужно отделить **проблему парсинга** от **проблемы LLM**:
- Если `*.txt` после PyPDF2 — мусор (склеенные колонки, потерянный порядок, пустота на сканах), то виноват парсер. **Действия:** пробуем `pdfplumber` или `PyMuPDF` как замену PyPDF2; если и они ломаются — рассматриваем возврат к multimodal-подаче (Qwen3-VL) как fallback для «сложных» PDF.
- Если текст извлёкся чисто, а модель всё равно галлюцинирует или валит JSON — проблема в LLM/промпте. **Действия:** идём с цифрами на серьёзный разговор о пересмотре подхода (другая модель, few-shot, двухпроходный промпт).

---

## 12. Что НЕ делаем в этом спринте

- Не сравниваем модели.
- Не считаем precision/recall.
- Не делаем eval-харнес.
- Не размечаем golden set (5 резюме можно прочесть глазами быстрее, чем разметить).
- Не трогаем заметки рекрутера, граф, retrieval — это всё после.
- Не оптимизируем промпт. Один прогон, базовый промпт.

Все эти штуки — в полноценном R&D, который запускаем по результатам smoke-test'а.
