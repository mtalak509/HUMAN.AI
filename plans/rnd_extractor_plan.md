# R&D-спринт: smoke-test LLM-экстрактора

*Версия 0.2. Минимальная проверка концепции на 5 резюме через мультимодальный Qwen3-VL по OpenRouter. Это не go/no-go по Г1 из `core_architecture.md`, а нулевая итерация: посмотреть, на что вообще способен подход. Полноценный R&D с замером метрик — отдельно, если smoke-test даст зелёный сигнал.*

---

## 1. Цель

Ответить на один вопрос: **способен ли мультимодальный Qwen3-VL, получая на вход PDF-резюме как изображения, вернуть валидно структурированную информацию о кандидате в нашей Pydantic-схеме?**

Это «работает / не работает», без точных метрик. На 5 резюме статистика всё равно шумная, и не в ней смысл — смысл в том, чтобы получить руками потрогать выход модели и понять, в какую сторону копать дальше.

---

## 2. Что упрощаем по сравнению с полноценным R&D

|----------|-----------------|------------|

## 3. Корпус

**5 резюме в PDF.** Распределение:
- 3 шт. — обычные HH-выгрузки или Word-резюме.
- 1 шт. — самописный PDF с нестандартным layout.
- 1 шт. — «творческое» (длинное, двухколоночное, с визуальными элементами).

DOCX в этом спринте не используем — упрощаем подачу до одного формата.

Источник — публичные резюме с HH или личные знакомые с разрешением. Обезличивать не обязательно, файлы локально и нигде не публикуются.

---

## 4. Extraction schema

Pydantic v2, минимальный набор:

```python
from pydantic import BaseModel, Field
from typing import Optional, Literal

class Contact(BaseModel):
    type: Literal["email", "phone", "telegram", "linkedin", "other"]
    value: str

class Experience(BaseModel):
    from_date: str = Field(description="YYYY-MM или YYYY")
    to_date: Optional[str] = Field(description="None если по настоящее время")
    company: str
    role: str
    description: Optional[str]
    skills_mentioned: list[str]

class Education(BaseModel):
    institution: str
    degree: Optional[str]
    field: Optional[str]
    from_date: Optional[str]
    to_date: Optional[str]

class Resume(BaseModel):
    full_name: str
    contacts: list[Contact]
    experiences: list[Experience]
    education: list[Education]
    skills: list[str]
```

## 5. Модель и подача документа

**Модель:** `qwen/qwen3-vl-...` через OpenRouter (точное имя модели — на момент имплементации, OpenRouter совместим с OpenAI Chat Completions API).

**Подача PDF:**
1. PDF → растеризация в PNG постранично (`pdf2image` или `PyMuPDF`).
2. Каждая страница → base64.
3. Все страницы одного резюме отправляются одним запросом как массив `image_url` контента + текстовый промпт.

```python
content = [
    {"type": "image_url", 
     "image_url": {"url": f"data:image/png;base64,{page1_b64}"}},
    {"type": "image_url", 
     "image_url": {"url": f"data:image/png;base64,{page2_b64}"}},
    {"type": "text", "text": PROMPT}
]
```

**Настройки:** `temperature=0`. Structured output — пробуем через `response_format={"type": "json_schema", ...}` если OpenRouter+Qwen поддерживают; fallback на JSON mode + Pydantic-валидация с одним retry на невалидном выходе.

## 6. Промпт

Один промпт, zero-shot:

```
Ты извлекаешь структурированную информацию из резюме кандидата. 
На входе — изображения страниц PDF-резюме.

Заполни поля JSON-схемы согласно содержимому. 
Правила:
- Если поле не упомянуто в резюме — null.
- Не выдумывай данные, которых нет в тексте.
- Для текущего места работы to_date = null.
- skills_mentioned для каждой роли — только навыки, явно упомянутые 
  в описании этой конкретной роли (не сводный список).
- Сводный список skills — все навыки, которые встречаются в резюме.
```

---

## 7. Что проверяем глазами

На 5 резюме автоматические метрики бессмысленны. Вместо них — чек-лист на каждое резюме:

- [ ] Ответ модели — валидный JSON, парсится в `Resume`.
- [ ] `full_name` совпадает с резюме.
- [ ] Все контакты из резюме присутствуют в `contacts`.
- [ ] Количество `experiences` совпадает с резюме (±1).
- [ ] У каждого experience корректные даты, компания, роль.
- [ ] `description` не пустой там, где он был в резюме.
- [ ] `skills_mentioned` для каждой роли — реально упомянутые, не выдуманные.
- [ ] `skills` (сводный) разумный по объёму.
- [ ] Нет галлюцинаций (выдуманные навыки, компании, периоды).

Параллельно — для каждого резюме записываем стоимость в токенах и время отклика.

## 8. Pipeline

### День 1: подготовка (2–3 часа)
- Зарегистрироваться на OpenRouter, получить API-ключ, положить 5–10 $ на счёт.
- Собрать 5 PDF-резюме.
- Создать структуру проекта (минимальную, см. §9).
- Реализовать `pdf_to_images()` и проверить на одном резюме.

### День 2: имплементация и прогон (3–4 часа)
- Pydantic-схема.
- OpenRouter-клиент с подачей multimodal.
- Промпт.
- Прогон 5 резюме, сохранить выходы в `results/`.

### День 3: анализ (2–3 часа)
- Пройти чек-лист по каждому из 5 резюме.
- Записать findings в `smoke_test_findings.md`: что работает, что нет, какие категории ошибок встречаются.
- Принять решение по развилке (§11).

## 10. Артефакты на выходе

## 11. Развилки

### Зелёный: всё в целом работает
По чек-листу 80%+ галочек на большинстве резюме, нет систематических галлюцинаций. **Действия:** разворачиваем полноценный R&D (30 резюме, 3 модели, метрики) — у нас есть подтверждение, что копать стоит.

### Жёлтый: работает на простых, ломается на сложном
Простые HH-резюме читаются, творческие — нет. **Действия:** добавляем в полноценный R&D сегмент по типу резюме, проверяем, не помогает ли смена промпта на двухпроходный (сегментация → экстракция). Сам подход не отбрасываем.

### Красный: галлюцинации, schema compliance < 80%, или просто бред
Мультимодальный Qwen3-VL не справляется с задачей. **Действия:** проверяем альтернативу — text-only подача (PDF → текст через `pdfplumber` → text-only Qwen или GPT-4o-mini). Если и text-only ломается — продуктовая гипотеза под большим вопросом, идём с цифрами на серьёзный разговор о пересмотре подхода.

---

## 12. Что НЕ делаем в этом спринте

- Не сравниваем модели.
- Не считаем precision/recall.
- Не делаем eval-харнес.
- Не размечаем golden set (5 резюме можно прочесть глазами быстрее, чем разметить).
- Не трогаем заметки рекрутера, граф, retrieval — это всё после.
- Не оптимизируем промпт. Один прогон, базовый промпт.

Все эти штуки — в полноценном R&D, который запускаем по результатам smoke-test'а.
