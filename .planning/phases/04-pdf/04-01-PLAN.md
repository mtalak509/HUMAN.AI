<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → parser | PDF file path/bytes are untrusted input crossing into pypdf and the storage layer |
| parser → disk | filenames and paths are written under storage_root |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Denial of Service | PyPdfBackend.extract (malformed PDF / decompression bomb) | mitigate | Validate `.pdf` suffix + file exists before reading (V5); pypdf has internal depth/size limits. A hard max-file-size check is a recommended follow-up but full byte read is bounded by local disk; flagged, not blocking for local v1. |
| T-04-02 | Tampering / EoP | storage write path derived from filename | mitigate | Sanitize original filename with `Path(pdf_path.name).name` — strip any directory components; never use full untrusted path in the storage layout. |
| T-04-03 | Tampering | directory name derived from document_id | accept | document_id is SHA-256 hex (chars 0-9a-f only) — structurally safe as a directory name; no traversal possible. |
| T-04-04 | Info Disclosure | empty/garbled extraction silently treated as success | mitigate | extraction_status="empty" (D-08) + logger.warning on empty pages; downstream sees the flag rather than a false "ok". |

No high-severity unmitigated threats. T-04-01 max-file-size hardening noted as a non-blocking follow-up for the API layer (Phase 7), where untrusted uploads actually enter.
</threat_model>
