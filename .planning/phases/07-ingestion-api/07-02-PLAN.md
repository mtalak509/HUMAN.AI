<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| public client -> POST /documents | untrusted multipart upload (bytes + filename + content-type) crosses here |
| client -> GET /documents/{id} | untrusted path param document_id |
| API handler -> Neo4j / Celery | enqueue + status writes cross into infra |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-06 | Denial of Service | POST /documents upload size | mitigate | MAX_UPLOAD_BYTES=10MiB cap -> 413; reject empty -> 400. Bounds memory/disk per request. |
| T-07-07 | Tampering | uploaded filename (path traversal) | mitigate | `safe_name = Path(file.filename).name` before any disk write; storage dir is document_id (hex). No user string reaches a filesystem path unsanitized. |
| T-07-08 | Spoofing | non-PDF masquerading as PDF | mitigate (partial) | Validate .pdf suffix + content-type at the route -> 415. Deep content sniffing deferred (T-04-01 known follow-up); parser tolerates garbage by failing the task (failed_stage=parse), never crashing the API. |
| T-07-09 | Tampering | client-supplied document_id in GET | mitigate | GET document_id is only used as a bound $param in MATCH (no Cypher injection); it is never used to build a filesystem path. POST never trusts a client id — it derives id from bytes. |
| T-07-10 | Denial of Service | POST flooding (rate) | accept | No rate limiting in local v1 (single-user MVP; no auth layer). Documented as deferred — revisit before any public deployment. |
</threat_model>
