<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Celery broker (Redis) -> worker | task args (document_id) cross from the queue into the worker |
| worker -> Neo4j | status/graph writes cross into the database |
| stored file -> parser | the on-disk PDF is read and parsed by the worker |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | Cypher in status.py / tasks.py | mitigate | All Cypher uses bound $params (no f-string interpolation); Document MERGE only on .id. Grep gate forbids string-built Cypher. |
| T-07-02 | Spoofing | document_id task arg | mitigate | document_id is a 64-char SHA-256 hex; the task re-derives it via PdfParser from the actual bytes (parse re-computes SHA-256) — a forged id cannot point the parser at someone else's content. |
| T-07-03 | Denial of Service | Neo4j unavailable mid-pipeline | mitigate | Task raises on is_connected=False instead of silently "succeeding" with no write (CONTEXT ⚠) — failures are observable, not masked. |
| T-07-04 | Info disclosure | error text stored on Document | accept | error is str(e)[:2000]; resume PDFs are not secrets in local v1, no api_key is ever in the exception path (extractor T-05-05 already strips it). Truncation bounds storage. |
| T-07-05 | Repudiation | which stage failed | mitigate | failed_stage (parse|extract|write) recorded on Document so a failure is attributable to a step (D-06). |
</threat_model>
