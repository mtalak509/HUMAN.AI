# AI-память компании о кандидатах — архитектура ядра

*Версия 0.1 (working draft). Документ описывает только ядро: графовое хранилище, парсер, LLM-экстрактор и augmented generation. Юридический контур (152-ФЗ), монетизация, фронтенд, безопасность, e-mail реконтакт и второй ATS-коннектор сознательно вынесены за скобки.*

---

## 1. Контекст и скоуп

### Что строим
Backend-ядро Talent Intelligence Platform: пайплайн, который превращает резюме и заметки рекрутеров в графовую память компании, и retrieval-слой, который по запросу на естественном языке возвращает обоснованный шортлист.

### Что НЕ строим в этой итерации
- Фронтенд (только минимальный демо-UI или Postman-коллекция).
- 152-ФЗ контур, согласия, политики хранения.
- Биллинг, мультитенантность, RBAC.
- E-mail реконтакт.
- Транскрипты интервью, мессенджеры, внешний сорсинг.
- Второй ATS-коннектор (после Huntflow).
- Прод-grade observability, бэкапы, HA.

### Целевые показатели пилота
- 1 пилотный клиент, ~300 резюме.
- 1 рекрутер-пользователь.
- SaaS-развёртывание на нашей инфре.
- Бюджет на инфру минимизирован.

## 2. Архитектура ядра (high-level)

Поток данных — слева направо в двух плоскостях: **ingestion** (источник → граф) и **retrieval** (запрос → шортлист).

```
Источники         Парсер           LLM-экстрактор       Entity Resolver
(Huntflow,    →   (PDF/DOCX,   →   (structured       →  (дедуп,
файлы)            Huntflow JSON)   output, schema)       merge)
                                                              │
                                                              ▼
                                                        ┌─────────────┐
                                                        │  Neo4j      │
                                                        │  (граф)     │
                                                        │  ↕          │
                                                        │  Qdrant     │
                                                        │  (векторы)  │
                                                        └─────────────┘
                                                              ▲
                                                              │
NL-запрос    →    Query Planner →   Гибридный поиск    →  Grounded
рекрутера         (LLM, function    (Cypher hard +         Generation
                  calling)          Qdrant soft)           (LLM + provenance)
                                                              │
                                                              ▼
                                                          Шортлист
                                                          с обоснованием
```

Ключевой архитектурный приём: **граф — источник правды, векторное хранилище — индекс для нечёткого поиска**. Любой результат векторного поиска в Qdrant завершается переходом в Neo4j за полным контекстом узла. Это базовый паттерн KAG: вектор находит, граф объясняет.

## 3. Технологический стек

| Слой | Технология | Почему |
|------|-----------|--------|
| Язык / runtime | Python 3.11+ | Весь AI-стек тут, команда уже там |
| Web-фреймворк | FastAPI | Async из коробки, OpenAPI, Pydantic-нативно |
| Графовая БД | Neo4j Community 5.x | Зрелая Cypher-экосистема, опыт команды |
| Векторное хранилище | Qdrant | Опыт команды, фильтрация по payload, scaffolding для гибридного поиска |
| Очередь задач | Celery + Redis | Минимальный стек для async ingestion, расширяется до Airflow в будущем |
| Эмбеддинги | BGE-M3 (локально, CPU) | Мультиязычность, ru-качество, считаем один раз, не зависим от API |
| LLM-провайдер | API (выбор отложен) | Решение по экономике после первых замеров на корпусе |
| Object storage | Локальный диск → MinIO/S3 | Хранение исходников документов, миграция тривиальна |
| Валидация / схемы | Pydantic v2 | Всё — от extraction schema до API контрактов |
| Парсинг документов | pdfplumber, pypdf (fallback), python-docx, tesseract (OCR) | Каскад от лучшего к самому толерантному |
| Тесты / eval | pytest, собственный eval-харнес | Метрики экстрактора и retrieval катятся при каждом изменении |
| Деплой | Docker Compose на одной VM | Минимальная стоимость, хватит надолго |

**Вне MVP, но архитектурно учтено:** Airflow (оркестрация), MinIO (object storage), Neo4j Enterprise/AuraDB (если упрёмся в Community), отдельные read-replicas Qdrant.

## 4. Онтология графа

### 4.1. Принципы

- **Минимальный набор узлов**, который покрывает то, что мы реально извлечём из резюме и заметок на старте. Расширение — по факту боли.
- **Факт — отдельный узел.** Каждое утверждение о кандидате (навык, период работы, образование) висит как нода `Fact`, связанная с источником `Document`. Это удваивает количество узлов, но даёт провенанс, версионность и разрешение конфликтов из коробки.
- **Никаких свойств-строк там, где должен быть узел.** `Skill: "Python"` — это узел, а не строка в массиве у кандидата. Иначе нормализация невозможна.

### 4.2. Типы узлов (стартовый набор)

| Узел | Описание | Ключевые свойства |
|------|----------|-------------------|
| `Candidate` | Кандидат | id, full_name, status, created_at |
| `Contact` | Контакт кандидата | type (email/phone/telegram), value |
| `Skill` | Нормализованный навык | name, canonical_name, category |
| `Role` | Должность / позиция | title, canonical_title, seniority |
| `Company` | Компания-работодатель | name, canonical_name, industry |
| `Experience` | Период работы кандидата в компании на роли | from, to, is_current |
| `Education` | Запись об образовании | institution, degree, field, from, to |
| `Vacancy` | Вакансия | id, title, opened_at, closed_at, status |
| `Status` | Статус кандидата в воронке вакансии | name (rejected, in_progress, offered, hired, withdrawn), changed_at |
| `HRNote` | Заметка рекрутера или нанимающего | author, text, created_at |
| `Document` | Исходный документ-источник | type (resume/note/ats_field), file_uri, ingested_at |
| `Fact` | Атомарный извлечённый факт | predicate, value, confidence, model_version, extracted_at |

`Company` явно нужна, потому что без неё `Experience` не имеет смысла. Добавил, хотя в исходном списке её не было.

### 4.3. Ключевые связи

```
(Candidate)-[:HAS_CONTACT]->(Contact)
(Candidate)-[:HAS_EXPERIENCE]->(Experience)-[:AT_COMPANY]->(Company)
(Experience)-[:AS_ROLE]->(Role)
(Candidate)-[:HAS_EDUCATION]->(Education)
(Candidate)-[:HAS_SKILL]->(Skill)            // прямая связь — для скорости поиска
(Candidate)-[:APPLIED_TO]->(Vacancy)
(Candidate)-[:REACHED_STATUS {at}]->(Status)-[:IN_VACANCY]->(Vacancy)
(HRNote)-[:ABOUT]->(Candidate)
(HRNote)-[:IN_VACANCY]->(Vacancy)            // если есть контекст вакансии

// Провенанс — пронизывает всё
(Candidate)-[:HAS_FACT]->(Fact)
(Fact)-[:EXTRACTED_FROM]->(Document)
(Fact)-[:SUPPORTS]->(Skill | Experience | Education | …)  // факт стоит за конкретной связью или узлом
```

### 4.4. Логика провенанса (вариант А, факт-узел)

При извлечении мы не пишем «у кандидата есть навык Python» как просто `(:Candidate)-[:HAS_SKILL]->(:Skill {name:"Python"})`. Мы пишем:

1. Создаём `Fact {predicate: "has_skill", value: "Python", confidence: 0.9, model_version: "ext-v1.2", extracted_at: ...}`.
2. Связываем `Candidate -[:HAS_FACT]-> Fact -[:EXTRACTED_FROM]-> Document`.
3. Создаём/находим узел `Skill {name: "Python"}`.
4. Связываем `Fact -[:SUPPORTS]-> Skill` и параллельно `Candidate -[:HAS_SKILL]-> Skill` (денормализация для скорости).

При появлении конфликтующего факта — старый не удаляется, новый помечается актуальным (`Fact {is_current: true}`). История остаётся в графе.

Минус подхода: на 300 резюме граф вырастет до десятков тысяч узлов вместо тысяч. Это допустимо. На 100к кандидатов потребуется индексирование и, возможно, partitioning, но решать это будем по факту.

## 5. Компоненты ядра

### 5.1. Парсер

**Задача:** превратить любой источник (PDF, DOCX, JSON из Huntflow) в нормализованный объект `RawDocument` с plain text + метаданными.

**Каскад извлечения текста:**
1. PDF → `pdfplumber` (умеет колонки и таблицы).
2. Если результат пустой / шумный → `pypdf` (грубее, но проще).
3. Если в PDF нет текстового слоя → OCR через `tesseract` (`pytesseract` обёртка, ru+en языковые пакеты).
4. DOCX → `python-docx`.
5. Huntflow JSON → прямой маппинг полей в `RawDocument` без LLM-парсинга.

**Что парсер НЕ делает:**
- Не сегментирует на «Опыт работы» / «Образование» — это работа экстрактора, у него больше контекста.
- Не извлекает сущности — это работа экстрактора.

**Что парсер обязан сохранить:**
- Исходный файл в object storage (на старте — локальный диск, путь `/storage/documents/{document_id}/{original_filename}`).
- Извлечённый текст рядом (`/storage/documents/{document_id}/text.txt`).
- Метаданные документа в Neo4j как узел `Document` с `file_uri`, `text_uri`, `parser_version`, `ingested_at`.

**Идемпотентность:** хэш содержимого файла (SHA-256) — если документ с таким хэшем уже есть, повторно не парсим.

### 5.2. LLM-экстрактор

**Задача:** превратить `RawDocument` в массив черновых фактов, готовых к записи в граф.

**Подход:** schema-guided structured output. Большая Pydantic-схема (= JSON Schema) описывает все типы извлекаемых сущностей и связей. На вход — текст + контекст (тип документа, дата, автор, известные ранее факты о кандидате, если кандидат уже есть). На выход — массив `ExtractedFact` объектов с confidence от модели.

**Эмпирическое решение по типу structured output.** На smoke-тесте из 5 резюме (Qwen3 через OpenRouter, `rnd/smoke_test_findings.md`) `response_format=json_object` + Pydantic-валидация + 1 retry показал 0 фактических ошибок и 0 галлюцинаций. `response_format=json_schema` дал 2 фактические ошибки на 5 (галлюцинация `to_date` на текущей работе; неверный `degree` для колледжа) и системную неполноту сводного `skills` на 4-5/5 резюме. Дефолтом для продакшен-экстрактора берётся `json_object` + Pydantic. `json_schema` — отдельная гипотеза для полноценного R&D, с обязательным post-processing `skills ∪= union(experiences[*].skills_mentioned)`.

**Декомпозиция:**
- Для резюме — один батч-вызов LLM на весь документ, structured output по полной schema.
- Для коротких HR-заметок — отдельный, более узкий промпт (типы сущностей сильно меньше: status, rejection_reason, salary_expectation, soft_signal).
- Для Huntflow JSON-полей — без LLM, прямой маппинг.

**Что extraction schema покрывает на старте:**
- Базовая информация о кандидате (ФИО, контакты).
- Список Experience (период + компания + роль + краткое описание + извлечённые навыки на этой роли).
- Список Education.
- Сводный список Skills.
- Желаемая зарплата / роль (если упомянуто).

**Версионирование:** каждый запуск экстрактора пишет `model_version` в извлечённые факты. Это даёт возможность позже фильтровать «факты, извлечённые старой моделью» и пере-извлекать (когда решим, что пора — на MVP не пере-извлекаем).

**Eval:**
- Золотой набор: 50 резюме + 50 заметок, размеченных вручную в той же schema.
- Метрики: precision / recall по каждому типу сущности (Skill, Experience, Education, Contact).
- Eval запускается при каждом изменении промпта или модели; результаты пишутся в файл / простой dashboard.

### 5.3. Entity Resolver

**Задача:** при записи нового факта понять, относится ли он к уже существующему узлу в графе или к новому.

**Каскад матчинга для `Candidate`:**
1. Точный матч по email → слияние.
2. Точный матч по нормализованному номеру телефона → слияние.
3. Точный матч по ФИО + дате рождения → слияние.
4. Нечёткий матч: ФИО (нормализованное) + cosine similarity эмбеддинга резюме > порога → создать кандидата со статусом `pending_merge`, флаг для ручного подтверждения рекрутером.
5. Иначе — новый узел `Candidate`.

**Каскад для `Skill`:**
1. Точный матч по `canonical_name` (после простой нормализации: lowercase, trim, удаление версий) → использовать существующий узел.
2. Cosine similarity эмбеддинга названия в Qdrant > порога → использовать существующий узел.
3. Иначе — новый узел `Skill`.

**Каскад для `Company`:** аналогично Skill, плюс предпочтение к узлу с большим количеством связей (популярные имена).

**Hard-rule:** автоматический merge кандидатов разрешён только при **точном** матче (1–3). Нечёткий — всегда требует подтверждения. Цена ложного merge выше цены ложного дубля.

### 5.4. Graph Writer

Тонкий слой, который превращает массив `ExtractedFact` в Cypher-транзакции.

**Принципы:**
- Одна транзакция на документ. Если что-то падает — откатываем целиком, документ остаётся в очереди на повтор.
- Идемпотентность: повторный запуск writer'а на тех же фактах не создаёт дубликаты (используем `MERGE` по композитным ключам).
- Денормализация для retrieval: помимо `Candidate -[:HAS_FACT]-> Fact -[:SUPPORTS]-> Skill`, дублируем прямую связь `Candidate -[:HAS_SKILL]-> Skill` для производительности запросов.

### 5.5. Векторный слой (Qdrant)

**Что лежит в Qdrant:**

| Коллекция | Что эмбеддим | Назначение | Payload |
|-----------|--------------|------------|---------|
| `skills` | название навыка | нормализация при extraction | `neo4j_skill_id`, `canonical_name` |
| `companies` | название компании | нормализация | `neo4j_company_id`, `canonical_name` |
| `experiences` | текст одного периода работы (роль + описание + контекст) | soft-критерии в retrieval («банковский домен», «highload») | `neo4j_experience_id`, `candidate_id`, `from`, `to` |
| `resumes` | весь текст резюме | fallback «найди похожих на этого» | `neo4j_candidate_id` |

**Главное правило:** payload в Qdrant хранит ID соответствующих узлов в Neo4j. Qdrant не источник правды, это индекс. Любой поиск завершается hop'ом в Neo4j за контекстом.

**Эмбеддинги:** BGE-M3, локально на CPU. На 300 резюме это секунды; на 100к — десятки минут единоразово, потом инкрементально. Без GPU. Без зависимости от внешнего API.

### 5.6. Augmented Generation (KAG retrieval)

Симметрично экстрактору — три прохода.

**Pass 1 — Query Planner.**
- Вход: NL-запрос рекрутера (или структурированный запрос из шаблона вакансии в будущем) + описание онтологии графа в системном промпте.
- LLM с function calling. Tools: `filter_by_skill(skill, level?)`, `filter_by_experience(role?, company?, industry?, years?)`, `filter_by_status(status, vacancy_id?)`, `filter_by_rejection_reason(reason)`, `semantic_match(description)`, `exclude_hired(true)`, `limit(n)`.
- Выход: структурированный план с разделением hard-фильтров (точно в графе) и soft-критериев (векторно).

**Pass 2 — гибридный поиск.**
- Hard-фильтры → собираем Cypher-запрос → получаем кандидатный сет (от 5 до тысяч).
- Если кандидатный сет большой → soft-критерии превращаем в эмбеддинги, ищем в Qdrant с pre-filter по `candidate_id IN [...]`.
- Если хард-фильтров нет вовсе → семантический поиск по `experiences` или `resumes`, потом hop в граф.
- Финальное ранжирование: weighted score (графовая релевантность + векторная similarity + recency).

**Pass 3 — Grounded Generation.**
- Для топ-N кандидатов из ранжирования собираем обогащённый контекст из графа: имя, last_status, релевантные Experience, релевантные Skills, связанные Facts с источниками.
- LLM генерирует объяснение «почему подходит» в строгом формате: каждое утверждение должно содержать ссылку на `fact_id`.
- Постпроцессор валидирует ссылки. Утверждения без `fact_id` — отбрасываются. Это защита от галлюцинаций.
- Выход: `Shortlist[Candidate, score, reasoning_with_citations[]]`.

### 5.7. Async pipeline

Все ingestion-операции (parse → extract → resolve → write) — асинхронные, через Celery воркеры на Redis.

**API-контракт:**
- `POST /documents` → возвращает `document_id` и `task_id`, реальная обработка в фоне.
- `GET /documents/{document_id}` → текущий статус (`queued`, `parsing`, `extracting`, `resolving`, `written`, `failed`).
- WebSocket / SSE-канал на `task_id` — опционально для UI, на MVP можно polling'ом.

Retrieval-эндпоинт — синхронный (`POST /search` → шортлист), с таймаутом 10 секунд.

**Расширение в будущее:** Celery воркеры → DAG в Airflow. Архитектурно подготовлено: каждый шаг ingestion — отдельная идемпотентная операция, состояние полностью в Neo4j (нет in-memory state между шагами).

### 5.8. Сквозной слой: provenance и observability

Не блок, а правило, пронизывающее всё:

- Каждый узел и каждое ребро в графе должно быть прослеживаемо до `Document` через `Fact`.
- Каждый ответ retrieval должен содержать `fact_id` для каждого утверждения.
- Каждый запуск экстрактора и retrieval логирует: входы, выходы, версии моделей, latency, токены.
- Логи — в файл / простую таблицу в Postgres (если решим завести; для MVP — JSON-файлы).

Без этого через два месяца мы не сможем дебажить ни экстрактор, ни выдачу.

## 6. Roadmap (3–4 месяца)

Разбивка по фазам, каждая заканчивается проверяемым артефактом.

### Фаза 0. Подготовка (недели 1–2)

- Поднять окружение: Docker Compose с Neo4j, Qdrant, Redis, FastAPI-скелет.
- Закодировать онтологию как Pydantic-модели + Cypher-миграции (constraints, indexes).
- Залить в Neo4j 10–20 тестовых кандидатов «руками» через Cypher, чтобы проверить онтологию без всякого AI.
- Настроить eval-харнес (пустой, но рабочий: pytest + sample data).
- **Критерий выхода:** локально поднимается весь стек, в графе лежит модельный пример, базовые Cypher-запросы работают.

### Фаза 1. Парсер + экстрактор на одном резюме (недели 3–6)

- Парсер: PDF/DOCX, fallback каскад, OCR опционально.
- Pydantic extraction schema v1 (минимально полный набор сущностей по разделу 4.2).
- LLM-экстрактор v1: один вызов на резюме, structured output.
- Простейший Graph Writer без entity resolver (каждое резюме = новый кандидат, разрешаем дубли осознанно).
- Eval: разметить руками 20 резюме как золотой набор, замерить precision/recall.
- **Критерий выхода:** загрузил 5 резюме разного формата → в графе появились 5 кандидатов с experience, education, skills. Eval-метрики посчитаны.

### Фаза 2. Entity resolver + Huntflow + заметки (недели 7–10)

- Коннектор Huntflow: pull кандидатов, вакансий, статусов, заметок через API.
- Entity resolver: каскад для Candidate / Skill / Company.
- Qdrant: коллекции skills, companies, experiences. Эмбеддинги локально.
- LLM-экстрактор v2: отдельный промпт для HR-заметок.
- Расширить eval: заметки в золотом наборе.
- **Критерий выхода:** прогон по 300 резюме пилотного клиента + Huntflow-данные → граф наполняется без дублей, заметки превращаются в факты со статусами и причинами отказа.

### Фаза 3. KAG retrieval (недели 11–13)

- Query Planner: function calling, набор tools, описание онтологии в промпте.
- Гибридный поиск: Cypher hard-фильтры + Qdrant pre-filter + ранжирование.
- Grounded Generation с обязательными `fact_id` ссылками.
- Минимальный API-эндпоинт `POST /search`.
- **Критерий выхода:** на 5–10 исторических вакансиях рекрутер задаёт NL-запрос → получает шортлист с обоснованием → субъективная оценка «лучше или хуже моего ручного подбора».

### Фаза 4. Тюнинг и демо (недели 14–16)

- Прогон retrieval по N историческим закрытым вакансиям, замер метрик (см. гипотезы).
- Тюнинг промптов экстрактора и Query Planner по результатам eval.
- Минимальный демо-UI (Streamlit или Next.js на shadcn — несколько экранов).
- Подготовка демо-сценария для пилота.
- **Критерий выхода:** живая демонстрация на данных пилота.

### Что НЕ входит в эти 16 недель
- Промышленный фронтенд.
- 152-ФЗ, согласия, RBAC.
- Email-реконтакт.
- Мультитенантность.
- Бэкапы, мониторинг, алертинг.

## 7. Проверяемые гипотезы

Гипотезы ранжированы по риску. Если падает Г1 или Г2 — продукта в текущей форме нет.

### Г1. Качество LLM-экстракции на русскоязычных резюме и заметках достаточно для построения осмысленного графа

**Метрика:** precision ≥ 0.90 и recall ≥ 0.80 на ключевых типах (Contact, Experience, Skill, Education) на золотом наборе из 50 резюме + 50 заметок.

**Проверяется:** в фазе 1 для резюме, в фазе 2 для заметок.

**Что значит провал:** граф наполняется мусором, retrieval бесполезен. Митигация: смена модели, доработка промптов, дополнительная валидация на постпроцессинге, дробление больших промптов на специализированные.

### Г2. KAG retrieval на исторических вакансиях даёт лучший шортлист, чем pure vector search

**Метрика:** на N (10–20) исторических закрытых вакансий пилотного клиента считаем recall@10: попадает ли реально нанятый кандидат (или финалисты) в топ-10 шортлиста. Сравниваем KAG vs baseline (cosine similarity между описанием вакансии и резюме).

**Проверяется:** в фазе 4.

**Что значит провал:** дополнительная сложность графа не оправдана, проще делать векторный RAG. Митигация: пересмотр логики Query Planner, добавление структурных сигналов из графа (last_status, recency взаимодействия) в ранжирование.

### Г3. Entity resolver на пилотных объёмах даёт приемлемое качество без ручных доработок

**Метрика:** на 300 кандидатов клиента — precision слияния ≥ 0.99 (false-merge rate < 1%), recall слияния ≥ 0.85.

**Проверяется:** в фазе 2 на реальных данных пилота.

**Что значит провал:** либо дубли в графе, либо ошибочные слияния разных людей. Митигация: ужесточение порогов, перевод нечёткого матча в обязательное ручное подтверждение.

### Г4. Стоимость LLM-обработки одного кандидата экономически разумна

**Метрика:** стоимость полного цикла (parse + extract + resolve + write) одного резюме в рублях. Целевой ориентир: ≤ 5 ₽ на кандидата на старте.

**Проверяется:** в фазах 1–2 по факту.

**Что значит провал:** unit-экономика не сходится с любой моделью монетизации. Митигация: смена провайдера API, переезд на локальную модель, дробление на маленькую (классификация / разбор) и большую (полная экстракция) модели.

### Г5. Граф + векторный стек справляется с пилотным объёмом по latency

**Метрика:** retrieval-запрос (план → Cypher → Qdrant → grounded generation) укладывается в 10 секунд p95 на пилотных данных.

**Проверяется:** в фазе 3.

**Что значит провал:** UX неприемлем. Митигация: индексы в Neo4j, кеш Query Planner для частых запросов, распараллеливание Cypher и Qdrant фаз.

## 9. Резюме

Ядро продукта — это четыре блока (парсер, LLM-экстрактор, графовое хранилище Neo4j+Qdrant, KAG retrieval), пронизанные сквозным слоем provenance. Стек подобран минимальный, но архитектурно расширяемый: Celery → Airflow, локальный диск → S3, Neo4j Community → Enterprise — каждая миграция понятна и не требует переписывания ядра. На пилоте (300 резюме, 1 клиент) этот стек крутится на одной VM с Docker Compose.

Главные риски — качество LLM-экстракции на русскоязычных свободных текстах (Г1) и преимущество KAG над pure vector RAG на реальных вакансиях (Г2). Всё остальное — инженерия, которая решается ресурсами.
